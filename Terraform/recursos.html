Data sources

Referencian infraestructura existente. No crean nada.

aws_vpc.default data

AWS te entrega un VPC por defecto: una red virtual privada aislada de otros clientes. Todo lo que creamos (EC2, RDS) vive dentro de este VPC. No lo creamos nosotros porque ya existe; solo lo referenciamos para que Terraform sepa donde colocar los recursos y que red usar para configurar las reglas de firewall.

aws_subnets.default data

Las subnets son subdivisiones del VPC, cada una en una zona fisica distinta del datacenter (ej. us-east-1a, us-east-1b). AWS las reparte asi para que si una zona cae, la otra siga operando. RDS exige un grupo de al menos dos subnets para poder reubicarse si es necesario. La EC2 usa solo una.

aws_ami.al2023 data

Una AMI es una imagen de disco preconfigurada: el sistema operativo base que arranca la EC2 al encenderse. Buscamos la mas reciente de Amazon Linux 2023 (una distribucion de AWS basada en RPM, ligera y optimizada para EC2). No la creamos: la buscamos con most_recent = true para tener siempre la version mas parcheteada disponible.

Modulo security

Firewalls (Security Groups) que controlan que trafico entra y sale.

aws_security_group.ec2 recurso

Un Security Group es un firewall que se adjunta a un recurso: decide que trafico puede entrar y salir de el. Aqui abrimos dos puertos de entrada:

  • :22 (SSH) — para conectarnos por terminal y depurar.
  • :3000 (API) — para que los alumnos hagan curl a la API desde sus maquinas.

Y permitimos toda salida (egress a 0.0.0.0/0), porque la EC2 necesita descargar paquetes (dnf install), clonar el repo (git clone) y conectarse a la RDS.

Abrir a 0.0.0.0/0 significa "desde cualquier IP". En produccion esto seria un riesgo; en clase lo hacemos asi para que todos puedan probar sin restricciones.
aws_security_group.rds recurso

El firewall de la base de datos. A diferencia del de la EC2, no abre puertos al internet. Solo permite conexiones al :5432 (PostgreSQL) que vengan especificamente del Security Group de la EC2.

Esto significa que nadie desde internet puede llegar a la base de datos. Solo la EC2 —porque tiene el SG correcto asignado— puede conectar. Es el patron recomendado: la base queda inaccesible desde afuera, la API es el unico puente.

Modulo database

RDS PostgreSQL gestionada + password aleatorio.

random_password.master recurso

Genera un password aleatorio de 24 caracteres (sin simbolos, para no romper conexiones ni URLs). No es un recurso de AWS: es del provider random de Terraform. Lo usamos para no hardcodear la contrasena en el codigo ni tener que inventarla manualmente.

El password viaja a dos lugares:

  • La instancia RDS (como credencial del usuario api).
  • El user_data de la EC2 (para que la app pueda autenticarse).
aws_db_subnet_group.this recurso

RDS exige que le digas en que subnets puede ubicarse mediante un DB Subnet Group: un grupo logico de subnets (minimo dos, en zonas distintas). Si AWS necesita mover la base a otra zona por mantenimiento o fallo, tiene donde.

aws_db_instance.this recurso

La base de datos PostgreSQL gestionada por AWS. Configuramos:

  • db.t4g.micro — el tamano mas pequeno (free tier, ARM Graviton).
  • 20 GB gp3 — almacenamiento SSD de proposito general.
  • publicly_accessible = false — sin IP publica. Solo se alcanza desde dentro del VPC.
  • skip_final_snapshot = true — al hacer terraform destroy, no exige un backup final.
  • deletion_protection = false — permite borrarla con terraform destroy.
RDS es un servicio gestionado: AWS se encarga del parchado del motor, los backups, la replicacion y el reemplazo de hardware. Nosotros solo definimos el tamano, la version y las credenciales.

Modulo compute

La maquina virtual (EC2), sus claves SSH y la IP publica fija.

tls_private_key.this recurso

Genera un par de claves RSA de 2048 bits usando el provider tls de Terraform. La clave publica se sube a AWS (para que la EC2 la reconozca al hacer SSH) y la clave privada se guarda en el estado de Terraform y se escribe a un archivo .pem en tu maquina.

Esto evita tener que generar la clave manualmente con ssh-keygen y subirla por la consola. Terraform la crea y la distribuye por nosotros.
aws_key_pair.this recurso

Registra la clave publica en AWS con un nombre (users-api-key). Cuando creamos la EC2, le asignamos este key pair. Asi, al hacer ssh -i clave.pem ec2-user@IP, AWS valida que tu clave privada coincide con la publica registrada y te deja entrar.

aws_instance.this recurso

La maquina virtual. Configuramos:

  • AMI — la imagen de Amazon Linux 2023 (del data source).
  • t3.micro — 2 vCPUs, 1 GB RAM (free tier).
  • Subnet — la primera del VPC por defecto.
  • Security Group — el ec2-sg que abre :22 y :3000.
  • Key pair — para acceso SSH.
  • user_data — el script de arranque que instala todo y levanta la app.
aws_eip.this recurso

Una Elastic IP: una IP publica fija asociada a la EC2. Sin esto, cada vez que la instancia reinicie, AWS le asignaria una IP publica distinta y los curl y las conexiones dejarian de funcionar. La EIP no cambia: siempre apunta a la EC2.

local_file.private_key recurso

Toma la clave privada generada por tls_private_key y la escribe en un archivo clave.pem en tu maquina, con permisos 600 (solo tu usuario puede leerla — SSH exige esto o se niega a conectar). Asi tienes el archivo listo para hacer SSH sin pasos manuales.

user_data.sh.tpl (script de arranque)

Se ejecuta automaticamente la primera vez que la EC2 arranca (cloud-init). Terraform rellena los placeholders antes de mandarlo a AWS.

Paso Que hace Por que
dnf install -y git nodejs Instala git y Node.js La AMI viene limpia; sin esto no podemos clonar el repo ni correr la app
git clone / git pull Clona el repo si no existe; hace pull si ya esta Trae el codigo al primer arranque; en reinicios actualiza a la ultima version
cd /opt/app/Terraform/app Entra al directorio de la app El codigo Node vive en Terraform/app/ dentro del repo, no en la raiz
npm install --omit=dev Instala dependencias Express y pg no vienen con Node; hay que bajarlos
chown -R ec2-user Cambia el dueno de los archivos systemd ejecuta la app como ec2-user; si los archivos son de root, la app no puede leerlos
/etc/app.env Escribe credenciales de RDS + PGSSL=true La app lee PGHOST, PGPASSWORD, etc. del entorno. PGSSL=true activa SSL porque RDS force_ssl rechaza conexiones sin cifrar
systemd app.service Crea un servicio gestionado por systemd Si la app muere, systemd la reinicia en 3 segundos. Tambien arranca sola al reiniciar la EC2
systemctl enable --now Activa el servicio ahora y en cada arranque enable = arranca en cada boot; --now = arranca ya mismo

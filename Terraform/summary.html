Marco teorico — Terraform & IaC

Conceptos que necesitas saber antes de escribir una linea de Terraform

1 El problema: infra a mano

Manual (consola de AWS)

Haces clic tras clic para crear una EC2, una RDS, configurar security groups, abrir puertos...

IaC (Terraform)

Describes todo en archivos de texto. Un comando crea todo. Reproducible y versionable.

Configurar infraestructura a mano tiene problemas concretos:

  • No es reproducible — si necesitas el mismo entorno en otra region, empiezas de nuevo.
  • No hay control de versiones — nadie sabe que cambio, cuando, ni por que.
  • Propenso a errores — un clic de mas abre un puerto al mundo o borra una base.
  • Lento — crear 20 recursos a mano toma horas; con IaC, segundos.
2 Que es Infrastructure as Code

Infrastructure as Code (IaC) es describir tu infraestructura en archivos de texto que se pueden versionar, revisar y re aplicar las veces que haga falta. Obtienes el mismo resultado siempre.

En vez de "entrar a la consola y hacer clics", escribes:

  • "Quiero una EC2 con esta AMI y este tamano"
  • "Quiero una RDS Postgres con 20GB"
  • "Quiero un firewall que abra el puerto 3000"

Y Terraform se encarga de traducir eso en llamadas a la API de AWS. Si cometes un error, lo ves en el codigo antes de aplicarlo. Si necesitas otro entorno identico, copias los archivos y cambias la region.

3 Declarativo vs Imperativo

Imperativo

Le dices los pasos: "crea una VPC, despues una subnet, despues asociale un security group, despues..."

Declarativo (Terraform)

describes el estado final: "quiero una EC2 y una RDS conectadas". Terraform averigua como llegar ahi.

No le dices a Terraform como llegar al resultado. Le dices que quieres. Terraform compara lo que pides con lo que existe (gracias al state) y calcula el plan de accion. Si ya existe, no hace nada. Si falta, lo crea. Si sobra, lo borra.

4 El ciclo de vida de Terraform

El patron que repetiras toda la clase:

terraform init → terraform plan → terraform apply → terraform destroy
Comando Que hace
terraform init Descarga los providers (plugins de AWS, random, tls, etc.) y prepara el directorio. Se corre una vez al empezar y cada vez que agregas un provider o modulo nuevo.
terraform plan Muestra que crearia, modificaria o borraria. No cambia nada: es un dry-run. Siempre revisa el plan antes de aplicar.
terraform apply Ejecuta los cambios. Pide confirmacion (escribe yes). Tambien puedes saltarla con -auto-approve.
terraform destroy Borra absolutamente todo lo que Terraform gestiona. Para parar de facturar al terminar la clase.
5 Conceptos clave de Terraform
Concepto Que es Ejemplo en la clase
Provider Plugin que sabe hablar con un proveedor cloud aws (crea recursos en AWS), random (genera passwords)
Resource Algo que Terraform crea y gestiona aws_instance (la EC2), aws_db_instance (la RDS)
Data source Algo que ya existe y Terraform solo lee aws_vpc.default (el VPC por defecto de AWS)
Variable Entrada configurable del modulo aws_region, instance_type, db_name
Output Valor que Terraform devuelve tras aplicar api_url, ec2_public_ip, ssh_command
Module Carpeta que encapsula recursos relacionados para reutilizar modules/security, modules/database, modules/compute
State Archivo .tfstate: la memoria de Terraform Registra que creo, con que parametros, para poder saber que cambiar
6 Por que el state importa

El archivo .tfstate es la unica fuente de verdad de Terraform. Contiene el inventario de todo lo que creo, con sus IDs y propiedades reales.

Gracias al state, Terraform puede:

  • Saber que ya existe (y no crearlo de nuevo).
  • Saber que falta (y crearlo).
  • Saber que cambio en el codigo (y actualizarlo en AWS).
  • Saber que ya no esta en el codigo (y destruirlo).
Si borras el .tfstate, Terraform pierde toda memoria. Los recursos siguen existiendo en AWS (siguen facturando), pero Terraform no sabe que existen. Si corres plan, intentara crear todo de nuevo. Recuperarse requiere terraform import (manual y tedioso) o borrar todo a mano en la consola y empezar de cero.
En equipos, el state se guarda en S3 + DynamoDB (remoto, con lock). Asi nadie lo pisa ni lo pierde, y dos personas no aplican cambios al mismo tiempo.
7 Comandos mas utiles
Comando Para que sirve Cuando usarlo
terraform init Descarga providers y registra modulos Al empezar o tras agregar un provider/modulo
terraform fmt Formatea los archivos .tf (indentacion, alineacion) Antes de cada commit
terraform fmt -recursive Lo mismo pero incluyendo subcarpetas (modulos) Cuando tienes modulos
terraform validate Verifica que la sintaxis sea correcta (no conecta a AWS) Despues de escribir codigo, antes del plan
terraform plan Muestra que crearia/cambiaria/borraria (dry-run) Siempre, antes de apply
terraform plan -out plan.tfplan Guarda el plan en un archivo para aplicarlo despues En CI/CD o para revisar antes de confirmar
terraform apply Ejecuta los cambios (pide confirmacion) Cuando el plan te parece bien
terraform apply -auto-approve Aplica sin pedir confirmacion En CI/CD (cuidado en produccion)
terraform destroy Borra toda la infra gestionada Al terminar la clase (para no facturar)
terraform output Muestra los outputs definidos Para ver la IP, URL, comando SSH
terraform output -raw api_url Muestra un output sin comillas (para copiar/pegar) Para usar en curl o scripts
terraform state list Lista todos los recursos que Terraform gestiona Para ver que hay en el state
terraform show Muestra el contenido completo del state Para inspeccionar detalles de un recurso
terraform taint <recurso> Marca un recurso para ser destruido y recreado en el proximo apply Cuando quieres forzar la recreacion de algo
Flujo recomendado: fmt → validate → plan → apply. Siempre en ese orden.
8 Limitaciones de Terraform

Terraform es una herramienta poderosa, pero tiene limitaciones que conviene conocer:

  • El state es un punto unico de fallo. Si lo pierdes, Terraform pierde toda memoria de lo que creo. Los recursos siguen en AWS facturando, pero Terraform no los reconoce. Por eso en produccion se guarda en S3 con backup.
  • El state puede contener secretos. Passwords, claves privadas y credenciales se guardan en texto plano dentro del .tfstate. Nunca lo subas a git. En clase usamos .gitignore para excluirlo.
  • No hay rollback nativo. Si un apply falla a mitad de camino (creo la EC2 pero fallo la RDS), te quedas en un estado parcial. Terraform no deshace lo ya creado; tienes que corregir y volver a aplicar.
  • Logica limitada. Terraform no es un lenguaje de programacion completo. No hay if/else tradicional, ni manejo de errores con try/catch. Hay count, for_each y conditional, pero son limitados.
  • Drift manual. Si alguien cambia algo directamente en la consola de AWS (abrir un puerto, cambiar un tamano), Terraform no se entera hasta que corres terraform plan. No hay deteccion automatica de cambios externos.
  • Importar recursos existentes es manual. Si tienes una EC2 creada a mano y quieres que Terraform la gestione, tienes que importarla uno por uno con terraform import. No hay escaneo automatico.
  • Infra grande = plan lento. Terraform consulta el estado de cada recurso en AWS antes de calcular el plan. Con cientos de recursos, un plan puede tardar minutos.
  • Secretos en user_data son visibles. Como vimos en clase, el password de la RDS se inyecta en el user_data de la EC2, y cualquiera con acceso a la consola de AWS puede leerlo en base64. La mejora seria usar SSM Parameter Store + IAM Role.
Estas limitaciones no restan valor a Terraform: siguen siendo mucho mejores que gestionar infra a mano. Conocerlas te ayuda a anticipar problemas y decidir cuando complementar con otras herramientas (Packer, Ansible, CI/CD).
